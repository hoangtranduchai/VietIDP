# VietIDP Annotation Guidelines

These guidelines define how to label Vietnamese administrative documents for VietIDP evaluation. They are designed for local, de-identified research workflows and do not replace legal or privacy review.

## Output schema

Each document label JSON should include:

```json
{
  "filename": "example.pdf",
  "full_text": "optional full transcription if available",
  "extraction": {
    "loai_van_ban": "",
    "so_hieu": "",
    "ngay_ban_hanh": "",
    "co_quan_ban_hanh": "",
    "trich_yeu": "",
    "nguoi_ky": ""
  },
  "evidence": {
    "loai_van_ban": [{"page": 1, "text": "", "note": ""}],
    "so_hieu": [{"page": 1, "text": "", "note": ""}],
    "ngay_ban_hanh": [{"page": 1, "text": "", "note": ""}],
    "co_quan_ban_hanh": [{"page": 1, "text": "", "note": ""}],
    "trich_yeu": [{"page": 1, "text": "", "note": ""}],
    "nguoi_ky": [{"page": 1, "text": "", "note": ""}]
  }
}
```

Use an empty string for missing, illegible, or not-applicable values. Do not invent values.

Normative contract:
- `filename`, `extraction`, and `evidence` are required top-level keys.
- Every field listed in `extraction` must also appear in `evidence`.
- `evidence.<field>` must be a list.
- For non-empty fields, `evidence.<field>` must contain at least one item with `page`, `text`, and `note`.
- For empty fields, use either an empty list when there is no visible evidence, or one note-bearing item when the reason is known, such as `missing`, `stamp overlap`, `blurred`, `cropped`, or `not applicable`.
- Synthetic development fixtures generated by `scripts/generate_benchmark_data.py` follow the same extraction keys and include minimal synthetic evidence notes, but they are not a substitute for human dual annotation.

## Field definitions

### `loai_van_ban`

Definition:
- The document type/title class, usually centered and uppercase.

Examples:
- `Quyết định`
- `Thông báo`
- `Công văn`
- `Tờ trình`
- `Kế hoạch`

Rules:
- Normalize casing to title case where clear.
- Do not include the subject line after the document type.
- If the type is absent or illegible, use empty string.

### `so_hieu`

Definition:
- The official document number/code near `Số:` or `Số/Ký hiệu:`.

Examples:
- `123/QĐ-UBND`
- `45/TB-SYT`
- `01/2026/CV-ABC`

Rules:
- Preserve slashes, hyphens, periods, and alphanumeric codes.
- Do not include the literal prefix `Số:` unless it is part of the official code.
- If multiple document numbers appear, choose the primary header number.

### `ngay_ban_hanh`

Definition:
- The issue/signing date of the document, usually in the top-right header.

Examples:
- `2026-04-29`

Rules:
- Store dates as `YYYY-MM-DD` when day/month/year are known.
- Convert forms such as `ngày 29 tháng 04 năm 2026`, `29/04/2026`, and `29-04-2026`.
- If only month/year or year is visible, use empty string and note partial evidence.
- Do not use effective date unless it is clearly also the issue date.

### `co_quan_ban_hanh`

Definition:
- The issuing agency or organization in the upper-left header.

Examples:
- `Ủy ban nhân dân tỉnh Đồng Nai`
- `Sở Y tế Thành phố Hồ Chí Minh`
- `Trường Đại học Bách khoa`

Rules:
- Use the agency/organization that issued the document.
- Do not label the national motto (`CỘNG HÒA XÃ HỘI CHỦ NGHĨA VIỆT NAM`) as the issuing agency.
- If parent and child agencies both appear, include the most specific issuing unit unless the document clearly names the parent as issuer.

### `trich_yeu`

Definition:
- The subject/summary line describing what the document is about.

Examples:
- `Về việc phê duyệt kết quả lựa chọn nhà thầu`
- `Triển khai công tác tuyển sinh năm học mới`

Rules:
- Exclude the document type heading.
- Include the full subject line when it spans multiple lines.
- Preserve meaningful legal names, numbers, and dates.
- Do not summarize in your own words.

### `nguoi_ky`

Definition:
- The human signer name in the signature block.

Examples:
- `Nguyễn Văn A`
- `Trần Thị B`

Rules:
- Label the person name, not the title (`CHỦ TỊCH`, `GIÁM ĐỐC`) and not `(Đã ký)`.
- If the name is hidden by a stamp or unreadable, use empty string and mark evidence as illegible.
- If there are multiple signers, choose the primary signer for the main document; note additional signers in evidence.

## Null, missing, and illegible policy

Use empty string when:
- the field is not present,
- the field is present but unreadable,
- only a partial value is visible and cannot be reconstructed without guessing,
- the field is not applicable to the document type.

Annotators must not infer from external knowledge. If a value is likely but not visible, leave it empty and explain in `evidence.note`.

## Evidence policy

For each non-empty field, record evidence with:
- page number,
- copied visible text span,
- optional note for ambiguity.

When the value is empty because of illegibility or absence, evidence may include:
- page number,
- visible surrounding text,
- note such as `missing`, `stamp overlap`, `blurred`, `cropped`, or `not applicable`.

Bounding boxes are optional in this Package B protocol but should be added when layout tooling supports them.

## Dual annotation workflow

1. Annotator A labels the document independently.
2. Annotator B labels the same document independently.
3. A comparison script or reviewer identifies disagreements.
4. An adjudicator reviews only disagreements and produces the final label.
5. Final labels are frozen before manifest creation.

Disagreements should be classified as:
- OCR/readability ambiguity,
- field definition ambiguity,
- formatting normalization difference,
- annotator mistake,
- document genuinely missing the value.

## De-identification and local handling

For real administrative documents:
- remove or mask personal identifiers not required for the benchmark,
- keep raw files local unless legal/privacy review approves release,
- avoid committing raw documents, uploads, databases, OCR dumps, or extracted personal data,
- store publishable demo data separately from private benchmark data,
- maintain a mapping between raw and de-identified files only in a restricted local location.

Public release requires legal/privacy review. This project protocol supports privacy-aware research operations but does not certify compliance.

## Quality checks before manifest creation

Before building a manifest:
- every input has exactly one label JSON,
- every label JSON maps to an input file,
- date format is `YYYY-MM-DD` where known,
- no national motto is labeled as agency,
- no signer title is labeled as signer name,
- empty fields have evidence notes where possible,
- final labels are adjudicated and frozen.

Then run:

```bash
python scripts/build_dataset_manifest.py --input <dataset_dir> --labels <dataset_dir>/labels --output <dataset_dir>/manifest.json --dataset-name <name> --split <split> --source-type <source_type>
python scripts/audit_benchmark_artifacts.py --input <dataset_dir> --labels <dataset_dir>/labels --manifest <dataset_dir>/manifest.json
```
